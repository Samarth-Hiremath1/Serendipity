# Design Document: Serendipity Networking App

## Overview

Serendipity is a Next.js web application that uses AI to transform networking from a manual, pull-based process into an ambient, intelligent experience. The system aggregates SF networking events, generates personalized attendee recommendations with conversation starters, and maintains relationship continuity through automated contact logging and follow-up drafting.

The architecture prioritizes demo reliability through aggressive caching, pre-computation of AI results, and consistent seed data. All AI operations use Google's Gemini API with responses cached in Supabase to stay within free tier limits.

### Technology Stack

- **Frontend**: Next.js 14+ with App Router, React 18, Tailwind CSS
- **Backend**: Next.js API Routes (serverless functions)
- **Database**: Supabase (PostgreSQL 15+ with pgvector extension)
- **Authentication**: Supabase Auth (email/magic link)
- **AI/ML**: Google Gemini API (gemini-2.5-flash-lite model)
- **Embeddings**: Google text-embedding-004 (same API key)
- **Vector Search**: Supabase pgvector for semantic similarity
- **Deployment**: Vercel (Hobby tier)

### Data Flow

1. **User Onboarding**: User submits profile → API generates embedding → Store in Supabase
2. **Event Discovery**: Fetch events from sources → Calculate relevance scores → Cache results → Display ranked feed
3. **Intel Generation**: User clicks event → Generate attendee personas (if needed) → Calculate similarity → Generate intel cards → Cache all → Display
4. **Contact Logging**: User submits notes → Parse with Gemini → Generate follow-up draft → Store in Supabase
5. **All AI Calls**: Check cache first → If miss, call Gemini → Store response → Return result

## Components and Interfaces

### Frontend Components

#### Pages

**`/` (Landing Page)**
- Hero section with value proposition
- CTA button to sign up
- No authentication required

**`/onboard` (Onboarding)**
- Multi-field form for profile creation

- Event header: name, date, location, description
- Room summary section (optional, generated by AI)
- Top 3-5 recommended attendees as IntelCard components
- Full attendee list below (collapsed by default)

**`/contacts` (CRM Dashboard)**
- List of ContactCard components
- Each card shows contact details, event met at, follow-up draft
- Copy button for follow-up message
- Sorted by follow-up date

**`/contacts/new` (Log Contact)**
- Textarea for freeform notes
- Submit button triggers parsing
- Shows parsed result for confirmation before saving

### Backend API Routes

**`POST /api/auth/signup`**
- Creates user account via Supabase Auth
- Returns session token

**`POST /api/profile`**
- Request: `{ name, role, company, current_work, looking_for[], can_offer[], interests[] }`
- Generates embedding from profile text
- Stores profile in `user_profiles` table
- Response: `{ profile_id, success }`

**`GET /api/events`**
- Query params: `user_id`
- Fetches events from external sources (with fallbacks)
- Calculates relevance scores for user
- Returns ranked event list with explanations
- All results cached in Supabase

**`GET /api/events/[id]`**
- Query params: `user_id`
- Fetches event details
- Generates attendee personas if not exist
- Calculates attendee relevance scores
- Returns event with top attendees

**`GET /api/intel/[event_id]`**
- Query params: `user_id`
- Generates intel cards for top attendees
- Returns array of intel cards
- All results cached

**`POST /api/contacts`**
- Request: `{ user_id, event_id, notes }`
- Parses notes with Gemini
- Generates follow-up draft
- Stores contact in database
- Response: `{ contact_id, parsed_data, follow_up_draft }`

**`GET /api/contacts`**
- Query params: `user_id`
- Returns all contacts for user
- Sorted by follow-up date

### Gemini API Integration

**`/api/gemini/embed`**
- Generates embeddings using text-embedding-004
- Input: text string
- Output: 768-dimensional vector
- Cached by text hash

**`/api/gemini/generate`**
- Generates text using gemini-2.5-flash-lite
- Input: prompt + context
- Output: JSON or text response
- Cached by prompt hash

### Caching Layer

All AI responses stored in `ai_cache` table:
```typescript
interface AICache {
  id: string;
  cache_key: string;  // hash of request
  request_type: 'embedding' | 'generation' | 'parsing';
  request_params: object;
  response: object;
  created_at: string;
}
```

Cache lookup happens before every Gemini API call. Cache key is SHA-256 hash of `request_type + JSON.stringify(params)`.

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system—essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*


### Property 1: Profile validation rejects incomplete data

*For any* profile submission with one or more required fields empty or missing, the system should reject the submission and return a validation error.

**Validates: Requirements 1.4**

### Property 2: Valid profiles generate embeddings

*For any* valid profile with all required fields populated, the system should successfully generate a 768-dimensional embedding vector.

**Validates: Requirements 1.5**

### Property 3: Data persistence round-trip

*For any* data object (profile, event, attendee, contact, or follow-up draft), storing it in Supabase and then retrieving it should produce an equivalent object with all fields preserved.

**Validates: Requirements 1.6, 2.4, 3.5, 6.6, 7.7**

### Property 4: AI response caching prevents redundant calls

*For any* AI operation (embedding generation, event scoring, attendee generation, intel card generation, contact parsing, or follow-up drafting), performing the same operation twice with identical inputs should only call the Gemini API once, with the second request served from cache.

**Validates: Requirements 1.7, 2.8, 3.7, 4.5, 5.7, 6.8, 7.8**

### Property 5: Event relevance scoring completeness

*For any* user profile and any set of events, the system should calculate a relevance score (0-100) and generate a non-empty explanation for each event.

**Validates: Requirements 2.5, 2.6**

### Property 6: List sorting invariant

*For any* list of items with numeric scores (events by relevance score, attendees by relevance score, contacts by follow-up date), the displayed list should be sorted in descending order by the specified field.

**Validates: Requirements 2.7, 8.6**

### Property 7: Attendee generation produces valid personas

*For any* event without existing attendees, the system should generate exactly 8-10 attendees, where each attendee has a name, role matching SF tech archetypes (founder, PM, engineer, investor, operator), company, bio, and a 768-dimensional embedding.

**Validates: Requirements 3.2, 3.3, 3.4**

### Property 8: Attendee consistency (idempotence)

*For any* event, generating attendees multiple times should return the same set of attendees (same names, roles, companies, bios) across all requests.

**Validates: Requirements 3.6**

### Property 9: Top attendee ranking and filtering

*For any* event with attendees and any user profile, the system should return 3-5 attendees ranked by relevance score in descending order.

**Validates: Requirements 4.4**

### Property 10: Rendered output completeness

*For any* data object being displayed (event intel page, attendee card, contact card), all required fields specified in the requirements should be present in the rendered output.

**Validates: Requirements 4.2, 4.6, 8.3, 8.4**

### Property 11: Intel card generation completeness

*For any* top-ranked attendee, the generated intel card should contain three non-empty fields: why_meet (specific reason), conversation_starter (non-generic opener), and ask_or_offer (suggested action).

**Validates: Requirements 5.1, 5.2, 5.3, 5.4**

### Property 12: Contact parsing extracts required fields

*For any* valid freeform contact notes, the parser should extract all required fields: name, role, company, topics_discussed, commitments, and follow_up_date.

**Validates: Requirements 6.4**

### Property 13: Invalid input error handling

*For any* invalid input (incomplete profile, unparseable contact notes, malformed data), the system should return a descriptive error message and reject the submission without storing partial data.

**Validates: Requirements 6.5, 10.6**

### Property 14: Contact-event referential integrity

*For any* contact created from an event, the contact record should contain a valid event_id that references an existing event in the database.

**Validates: Requirements 6.7**

### Property 15: Follow-up draft generation and content

*For any* contact created, the system should automatically generate a follow-up draft that contains the event name, references topics discussed from the contact, and is 5 sentences or fewer.

**Validates: Requirements 7.1, 7.2, 7.3, 7.5**

### Property 16: CRM displays all user contacts

*For any* user with N contacts in the database, viewing the CRM dashboard should display all N contacts.

**Validates: Requirements 8.2**

### Property 17: Cache hit returns stored response

*For any* cached AI request, if a cache entry exists with a matching cache key, the system should return the cached response without calling the Gemini API.

**Validates: Requirements 9.2**

### Property 18: Cache miss stores new response

*For any* AI request without a cached response, the system should call the Gemini API and store the response in the cache table with a hash-based cache key.

**Validates: Requirements 9.3, 9.4**

## Error Handling

### API Error Handling

**Gemini API Errors:**
- Catch all Gemini API errors and log to console/monitoring
- Display user-friendly message: "AI service temporarily unavailable. Please try again."
- For rate limit errors (429), check cache first, then display: "Please try again in a few moments."
- Never expose raw API error messages to users

**Event Source Errors:**
- Implement fallback chain: Luma → Eventbrite → Meetup → Cached events
- Log which source succeeded/failed for monitoring
- If all sources fail and no cache exists, display: "Unable to load events. Please check your connection."

**Database Errors:**
- Catch Supabase connection errors
- Display: "Database connection error. Please refresh the page."
- Prevent partial writes with transactions where appropriate
- Log errors for debugging

### Input Validation

**Profile Form:**
- Validate all required fields are non-empty before submission
- Show inline error messages for each invalid field
- Disable submit button until form is valid

**Contact Notes:**
- Validate notes are non-empty before parsing
- If parsing fails, show: "Unable to parse notes. Please include name, role, and what you discussed."
- Allow user to edit and resubmit

**General Input Sanitization:**
- Sanitize all user inputs to prevent XSS
- Validate data types match expected schema
- Trim whitespace from text inputs

### Edge Cases

**Empty States:**
- No events available: Show message with retry button
- No attendees for event: Trigger generation automatically
- No contacts yet: Show empty state with CTA to log first contact
- No intel cards: Show loading state during generation

**Network Failures:**
- Implement retry logic with exponential backoff for transient failures
- Show loading states during API calls
- Timeout long-running requests after 30 seconds

**Concurrent Requests:**
- Use database transactions to prevent race conditions
- Handle duplicate event/attendee creation with UNIQUE constraints
- Use optimistic locking for updates where needed

## Testing Strategy

### Dual Testing Approach

The system will use both unit tests and property-based tests to ensure comprehensive coverage:

- **Unit tests**: Verify specific examples, edge cases, error conditions, and integration points
- **Property tests**: Verify universal properties hold across all inputs through randomization

Both testing approaches are complementary and necessary. Unit tests catch concrete bugs in specific scenarios, while property tests verify general correctness across the input space.

### Property-Based Testing Configuration

**Library**: Use `fast-check` for TypeScript/JavaScript property-based testing

**Configuration**:
- Minimum 100 iterations per property test (due to randomization)
- Each property test must reference its design document property
- Tag format: `// Feature: serendipity-networking-app, Property {number}: {property_text}`

**Test Organization**:
- Co-locate tests with source files using `.test.ts` suffix
- Group related properties in test suites
- Use descriptive test names that explain what property is being verified

### Unit Testing Focus

Unit tests should focus on:
- Specific examples that demonstrate correct behavior (e.g., "profile with all fields creates successfully")
- Integration points between components (e.g., "API route calls database correctly")
- Edge cases and error conditions (e.g., "empty profile fields return validation error")
- UI component rendering (e.g., "EventCard displays all required fields")

Avoid writing too many unit tests for scenarios that property tests already cover. Property tests handle comprehensive input coverage through randomization.

### Property Test Implementation

Each correctness property must be implemented as a single property-based test:

**Example Property Test Structure:**
```typescript
// Feature: serendipity-networking-app, Property 1: Profile validation rejects incomplete data
test('profile validation rejects incomplete data', () => {
  fc.assert(
    fc.property(
      fc.record({
        name: fc.option(fc.string(), { nil: '' }),
        role: fc.option(fc.string(), { nil: '' }),
        company: fc.option(fc.string(), { nil: '' }),
        // ... other fields
      }),
      (profile) => {
        const hasEmptyField = Object.values(profile).some(v => !v);
        if (hasEmptyField) {
          const result = validateProfile(profile);
          return result.isError && result.error.includes('required');
        }
        return true;
      }
    ),
    { numRuns: 100 }
  );
});
```

### Test Coverage Goals

- All 18 correctness properties implemented as property tests
- Unit tests for all API routes
- Unit tests for all React components
- Integration tests for critical user flows (onboarding → events → intel → contacts)
- Error handling tests for all external dependencies

### Testing AI Components

**Gemini API Integration:**
- Mock Gemini API responses in tests to avoid rate limits
- Test cache hit/miss logic with real database
- Verify prompt construction and response parsing
- Test error handling for API failures

**Embedding Generation:**
- Mock embedding vectors in tests (use fixed 768-dim arrays)
- Test vector similarity calculations with known inputs
- Verify embeddings are stored correctly in pgvector

**LLM Response Parsing:**
- Test with variety of valid and invalid LLM outputs
- Verify JSON parsing handles malformed responses
- Test fallback behavior when parsing fails

### Database Testing

- Use Supabase test project or local PostgreSQL with pgvector
- Reset database state between test runs
- Test all CRUD operations
- Verify indexes are used for vector similarity queries
- Test transaction rollback on errors

### End-to-End Testing

For demo reliability, create one comprehensive E2E test that simulates the full user journey:
1. Sign up and create profile
2. View event feed with relevance scores
3. Click event and view intel cards
4. Log a contact with notes
5. View contact in CRM with follow-up draft

This E2E test should run against a staging environment before the demo to verify all integrations work.
